# Architecture Decisions

本文件采用简化的 ADR（Architecture Decision Record）形式记录当前关键决策。

## ADR-001：使用单 Agent，而不是多 Agent

- 状态：Accepted
- 背景：项目第一版的核心问题是验证“动态调用工具 + 证据绑定 + 确定性评分”的最小可行链路，而不是组织多个角色协作。
- 决策：MVP 采用单 Agent 架构，统一负责决策与调用六个核心 Tool。
- 原因：单 Agent 更容易建立清晰的 State、工具轨迹和测试基线，能减少协调开销。
- 代价或后果：复杂场景下的任务拆分能力较弱；若后续扩展到多 Agent，需要重新设计协作协议和共享 State。

## ADR-002：使用 LangGraph 作为 Agent 编排框架

- 状态：Accepted
- 背景：JobPilot 需要显式管理 State、循环调用、终止条件和可观测轨迹。
- 决策：MVP 采用 LangGraph 作为 Agent 编排框架。
- 原因：LangGraph 适合表达带 State 的 Agent Loop，也便于后续加入节点级测试、分支控制和人工介入点。
- 代价或后果：框架抽象会带来额外学习成本；过于简单的流程也可能显得偏重。

## ADR-003：MVP 使用本地 Mock 数据，不直接访问真实招聘网站

- 状态：Accepted
- 背景：项目当前目标是验证架构和行为约束，而不是解决真实招聘网站接入、反爬和稳定性问题。
- 决策：第一版仅使用本地 `jobs.json` 和本地文件输入。
- 原因：有利于测试可重复性、离线开发和安全边界控制。
- 代价或后果：岗位覆盖有限，结果不代表真实市场全量信息；后续接入真实数据源时需要补充 Provider 与合规设计。

## ADR-004：LLM 负责语义判断，普通代码负责确定性评分

- 状态：Accepted
- 背景：LLM 擅长理解开放式文本，但不适合作为最终确定性评分器。
- 决策：由 LLM 负责语义理解、缺口判断和 Tool 选择，由普通代码负责岗位匹配评分与规则执行。
- 原因：这样既能保留 Agent 的灵活性，也能保证关键结果可重复、可测试。
- 代价或后果：需要维护一套明确的评分规则；当规则不合理时，需要人工迭代设计。

## ADR-005：所有简历结论必须绑定证据

- 状态：Accepted
- 背景：求职场景对事实准确性要求高，夸大能力会直接影响用户风险。
- 决策：任何写入候选人能力、经历、匹配说明或投递材料的结论，都必须绑定 `evidence_id`。
- 原因：保证可追溯性、可审计性，并为自动化测试提供断言基础。
- 代价或后果：输出生成更严格，可能减少文案流畅度；部分弱证据场景需要显式标记“证据不足”。

## ADR-006：前端只调用 FastAPI，不直接调用 Agent 工具

- 状态：Accepted
- 背景：如果前端直接调用 Tool，会绕过输入校验、权限边界和统一轨迹管理。
- 决策：Frontend 只通过 FastAPI 访问系统能力，不直接接触 Tool 实现。
- 原因：有利于维持安全边界、统一日志与状态管理，也更方便替换前端实现。
- 代价或后果：前端调试路径比直连 Tool 多一层；API 设计需要保持清晰稳定。
