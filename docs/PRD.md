# JobPilot PRD

## 1. 产品背景

JobPilot 是一个面向求职执行场景的智能 Agent 系统。这里的 Agent 指能够根据上下文自主决定下一步工具调用的执行体，而不是预先写死的固定流程。系统围绕用户的求职目标、简历、可选岗位 JD 和可选本地项目代码，决定是否搜索岗位、读取岗位详情、补充证据、评分并生成投递材料。

## 2. 问题定义

传统“简历对 JD”工具通常假设输入固定、流程固定，无法根据用户是否已提供 JD、简历证据是否充分、是否存在本地项目代码等情况动态调整。JobPilot 要解决的问题是：在信息完整度不一致的前提下，仍然以可追溯证据为基础，生成可靠的岗位匹配分析和投递材料。

## 3. 目标用户

- 已有明确求职目标，但材料尚未整理完成的求职者。
- 希望基于已有简历和本地项目代码补强岗位匹配证明的开发者。
- 需要可解释、可追踪的投递建议，而不是黑盒式推荐的用户。

## 4. 核心价值

- 根据输入状态动态选择路径，而不是强制执行固定 Workflow。
- 所有能力结论都绑定证据，降低夸大表述风险。
- 用确定性评分补足 LLM 在最终判断上的不稳定性。
- 为后续自动化测试、Golden Cases 和 HITL 扩展保留清晰边界。

## 5. 用户输入

- `user_goal`：开放式求职目标，例如岗位方向、城市、薪资偏好。
- `resume_text` 或 `resume_path`：本地简历文本或文件。
- `provided_jd`：用户直接提供的岗位 JD，可为空。
- `project_path`：本地项目代码目录，可为空。
- 可选偏好：关键词、行业、资历、工作方式等。

## 6. 系统输出

- 候选人画像 `candidate_profile`。
- 岗位候选列表 `job_candidates`。
- 已检查岗位详情 `inspected_jobs`。
- 证据台账 `evidence_ledger`。
- 岗位匹配报告 `fit_reports`。
- 最终投递材料包 `final_report`，包含事实化摘要、岗位匹配说明和后续建议。
- Agent 执行轨迹，用于调试、审计和测试。

## 7. 核心用户场景

### 场景 A：用户提供了完整 JD

用户给出求职目标、简历和完整 JD。系统应直接解析 JD、检查证据、计算匹配，不应再调用岗位搜索。

### 场景 B：用户未提供 JD

用户只给出求职方向和简历，且明确希望寻找或推荐岗位。系统应先搜索本地岗位库，再读取选中的岗位详情并继续匹配。

### 场景 B-1：用户未提供 JD，但只要求简历分析

用户没有提供 JD，且只要求解析简历、建立候选人画像或总结经历。系统不应触发岗位搜索，而应停留在简历理解和事实整理范围内。

### 场景 C：简历证据不足但有本地项目

系统发现简历无法证明某项关键要求时，可以检查本地项目代码，提取更细的技能或成果证据。

### 场景 D：简历与项目都无法证明能力

系统必须明确标注证据不足，不得把该能力写成已掌握；后续版本可暂停并向用户追问。

## 8. MVP 功能

- 单个求职 Agent。
- 本地简历文本或文件输入。
- 本地 `jobs.json` 岗位库搜索与读取。
- 本地项目代码证据检索。
- 确定性的岗位匹配评分。
- LangGraph Agent Loop。
- FastAPI 后端。
- Streamlit 前端。
- Agent 执行轨迹。
- 自动化测试与 Golden Cases。

## 9. 非目标

- 多 Agent 协作。
- 招聘网站爬虫或真实站点抓取。
- 真实 GitHub API 集成。
- 自动投递、自动发送邮件。
- 自动修改用户原始简历文件。
- 用户账户系统。
- 云端部署。
- 向量数据库。
- RAG 知识库。

## 10. 功能验收标准

以下标准必须可通过自动化测试验证：

- 当 `provided_jd` 存在且内容完整时，禁止调用 `search_jobs`。
- 当 `provided_jd` 缺失，且用户明确要求寻找或推荐岗位时，必须调用 `search_jobs`。
- 当 `provided_jd` 缺失，但用户只要求解析简历、建立画像或总结经历时，不应调用 `search_jobs`。
- 当简历证据不足且存在 `project_path` 时，允许调用 `inspect_project_evidence`。
- 当简历和项目都缺少关键证据时，禁止把相关技能写成已掌握或已验证。
- `score_job_fit` 必须输出结构化评分结果，且相同输入得到相同分数。
- `generate_application_pack` 只能使用已验证事实和已登记 `evidence_id`。
- Agent 工具调用必须有上限，不得无限循环。
- 执行轨迹必须记录每一步调用的工具名、输入摘要、输出摘要和状态。

## 11. 非功能需求

- 可测试性：核心路径必须支持 Golden Cases。
- 可追溯性：所有结论可回溯到 JD 或 `evidence_ledger`。
- 可解释性：最终报告需区分“已验证事实”“推断”“证据不足”。
- 可维护性：工具契约、State 结构、评分逻辑与 LLM 决策边界明确。
- 安全性：不默认访问外网，不修改用户原始简历，不泄露本地敏感内容。
- 性能：MVP 在本地单次执行中应有明确步数上限，避免无界循环。

## 12. 风险与限制

- LLM 对语义理解存在波动，因此最终评分不能完全交给模型。
- 本地项目证据检索可能找到“出现过的关键词”，但不等于真实掌握，需要证据分级。
- 简历文本质量参差不齐，候选人画像可能依赖补充提问。
- 仅使用本地岗位库时，岗位覆盖度有限，搜索结果不代表真实市场全量机会。
- MVP 不接入真实投递渠道，因此最终产物只用于人工复核与后续操作。
